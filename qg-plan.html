1 Методика отбора: что считается рабочим QG

Статус active/draft в LORE — не критерий (часть draft новее и полнее). Решаем по содержимому. QG в наборе, только если проходит все 5 фильтров:

  • Рекуррентность — повторяется по триггеру (merge / weekly / pre-demo / re-harvest), не разовый отчёт.
  • Проверяемость — детерминированный PASS/WARN/FAIL по инварианту, не описательный анализ.
  • Неизбыточность — один канонический гейт на домен; дубли сливаем.
  • Привязка к срезу — есть routine_name и место в одном из 5 срезов или в агрегате.
  • SMART-метрики — каждый INV даёт (key,value,unit,target,status,source) с file:line evidence.

5 Методика расчёта статуса

Статус метрики

def compute_status(value, target, direction="gte"):
    if value == -1:            # источник недоступен
        return "SKIP"
    if direction == "gte":
        if value >= target:        return "PASS"
        if value >= target * 0.9:  return "WARN"
        return "FAIL"
    else:  # lte
        if value <= target:        return "PASS"
        if value <= target * 1.1:  return "WARN"
        return "FAIL"
# dir "=" (HTTP): value==target → PASS, иначе FAIL (down→SKIP)
# unit bool: target=1, gte → 0=FAIL, 1=PASS

Статус всего прогона

FAIL≥1 метрика FAIL (сервис жив, инвариант нарушен)
WARNнет FAIL, но есть WARN или SKIP
PASSвсе метрики PASS

SKIP: сервис down (value=-1) → SKIP, не FAIL. Отделяет «у нас сломано» от «сервис был выключен».

Связи: routine_name = срез; last_run_status (active/blocked) ≠ lifecycle status; flags = summary + fallback для старых прогонов.

6 Как пишем рекомендации

INV FAIL→ QGJobTask (open)→ QGRecommendation (pending)→ подтверждение→ KnowTask (спринт)→ PASS → resolved

Rolling-QG не засоряют спринты — задача создаётся только после подтверждения (ADR-LORE-011). Точность даёт metric.source (file:line).

body_md рекомендации

секциясодержание
Root Causeпричина, не симптом
Evidencefile:line + число из metric.source
Impactчто сломано / рискует сломаться
ContextADR / инцидент / PR
fix_cmdкоманда/патч исправления
how_to_verifyкоманда, чей вывод докажет фикс

Пример: метрика → рекомендация

// метрика:
{ key:"inv2_safecall_count", value:5, target:8, status:"FAIL",
  source:"grep -n safeCall CompositeListener.java → lines 47,89,134 (=5, want 8)" }

// рекомендация:
title: "CompositeListener не изолирует 3 listener-вызова (safeCall 5/8)"
## Root Cause  invoke() оборачивает в safeCall 5 из 8 точек (4 метода × 2 listener).
## Evidence   safeCall на L47,89,134; ОТСУТСТВУЕТ в onFileStart(~L61), onRoutineEnd(~L102),
             onError(~L150). grep -c = 5, want 8.
## Impact     падение одного listener обрывает цепочку → остальные не получат событие.
## Context    ADR-HND-003: каждый listener-вызов изолирован.
fix_cmd:       обернуть 3 вызова в try{ l.invoke(e); }catch(Exception ex){ log.warn(...); }
how_to_verify: grep -c safeCall CompositeListener.java → 8; EventListenerIsolationTest зелёный

Без source=file:line было бы «safeCall FAIL» — неактионабельно. Поэтому в ClRoutineMetric добавлено поле source.
