3 Feature = мост (реифицированное ребро)

Фича — не «вещь, которую делаем», а утверждение: «для работы J, снимая боли P, создавая выгоды G». Ноги в профиль = ЗАЯВЛЕНО (фича обещает), ноги UC = СДЕЛАНО (сценарий реально снимает). Фича без ног в профиль — не продуктовая, это техдолг (jtd/enb-задачи, продуктовый слой их не видит).

4 Скоринг Остервальдера — из цифр на FEAT-GITCYCLE

Ранги на рёбрах привязки; скоринг вычисляется, не хранится. pain_score = severity(FELT_BY) × importance(блокируемой работы). Fit = закрытый вес / заявленный. Каждая цифра — в разрезе сегмента (у Агента свой fit, у Владельца свой).

5 Чек-лист принятых решений

R1
KnowProduct — новая вершина. Каталог продуктов AIDA; связывает ценность (канва), реализацию (фичи→UC), технику (компоненты, репо).
R2
Job/Pain — глобальные вершины (без BELONGS_TO_PROJECT); контекст продукта целиком на рёбрах.
R3
Gain — проектная вершина (BELONGS_TO_PROJECT), метрика привязана к продукту.
R4
Feature = мост: обязана иметь ≥1 ногу в профиль (ADDRESSES/PROMISES/HELPS_WITH), иначе не продуктовая.
R5
Ранги — на рёбрах привязки к сегменту (Q1): severity на FELT_BY, importance на PERFORMED_BY, relevance на DESIRED_BY. Вершинный severity — дефолт «для всех». Сегментов реально много → рёбра нужны сразу.
R6
Скоринг по-сегментный, вычисляется обходом рёбер; fit — в разрезе продукт × сегмент.
R7
Эффективность — бинарная пока (Q2): есть RELIEVES → закрыто. Та же формула даст градуированные % позже, когда поедет вычислитель shipped_at — без переделки.
R8
Ранги — 3 ступени (Q4): высокая/средняя/низкая; словари уже трёхступенчатые, «средняя» боль реально есть в данных.
R9
Скоринг в навигации (Q3): подсветка «горячих» в VP-канве + бэклог по feature_value в Аналитике. Отдельный «Приоритизатор» не нужен.
L1
Гипотезы / тестирование — ПОЗЖЕ. Сначала наполним карточки; проверять пока нечего, KnowFinding уже есть как крючок на потом. Сейчас не заводим.

6 Навигация «Зачем» (глава Storyline)

Порядок по Остервальдеру: кто → его профиль → сборка → ответ → реализация.
