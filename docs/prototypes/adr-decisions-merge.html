ADR-TS-001 ACCEPTED Протокол наружу (OTel/PromQL-ready) — UI внутрь (HEIMDALL). Свой observability-стек не разворачиваем OBS
Context — почему
Наблюдаемость требует стека: сбор → хранение → визуализация. Индустриальный дефолт — Prometheus + Grafana (+ Jaeger/Tempo). Здесь две разные задачи, которые легко спутать:
· наша эксплуатация — нам нужен рабочий UI, и он уже есть (HEIMDALL: in-process сборщики → TIMESERIES в FRIGG → Heimdall Capacity);
· интеграция у заказчика — у него может быть СВОЙ observability-стек, и он вправе забрать наши метрики к себе.
Разворачивать Grafana/Prometheus-сервер у себя — значит платить несколькими сервисами в compose и своей эксплуатацией за задачу, которая у нас уже решена.
Decision — что решили
Протокол — да, стек — нет.
· Наружу: остаёмся OTel/PromQL-ready — заказчик может подключить свою Grafana / свой OTel-коллектор (PromQL endpoint в ArcadeDB) и забрать данные без миграции.
· Внутрь: собственный UI наблюдаемости — HEIMDALL. Grafana / Prometheus-сервер / OTel-коллектор в наш compose не добавляются.

Consequences — следствия
Проверено 2026-07-10: в compose 0 сервисов Grafana / Prometheus-сервера / OTel-коллектора — ограничение выдержано за 4 месяца и 4 решения. Способ менялся (OTLP → Prometheus-плагин), ограничение — нет. Ровно то, ради чего ADR и существует: правило пережило смену реализации.

Ключевое, что «протокол наружу» — не отложенная задача, а действующее обязательство: `PromQL endpoint` держится ради заказчика, даже пока свою Grafana мы не ставим. Спутать эти два уровня — значит либо развернуть лишний стек у себя, либо закрыться от интеграции у клиента.

Один-ко-многим: один ADR-ограничение → 4 решения, принятых в разное время разными людьми. В плоской модели (ADR-016) эта связь была бы невидима — каждое решение стало бы отдельным ADR без общего родителя.
