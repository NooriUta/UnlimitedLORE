## 0. Текущее состояние (As-Is, факты)

**Модель:** граф ArcadeDB, ~12 бизнес-типов вершин + 8 парных `*Hist`-типов (SCD2) + ~36 типов рёбер. Запись — только через Quarkus-эндпоинты (`X-Seer-Role: admin`), чтение — через whitelisted-слайсы, LLM-доступ — MCP поверх HTTP.

**Ключевой паттерн — SCD2 с «открытой строкой»:** текущее состояние = `*Hist`-строка с `valid_to IS NULL`, связанная `HAS_STATE`. Статусные переходы закрывают/открывают строки (честная история); **правки тел мутируют открытую строку по месту** (истории тел нет).

**Распределение полей vertex ↔ hist — несистемное, по типам:**

| Тип | В hist (слайс читает оттуда) | В вершине | Комментарий |
|---|---|---|---|
| ADR | context/decision/consequences_md | name, status, date, file_path | status на вершине = денорм, честные переходы в hist |
| Spec | content_md, version, summary (COALESCE→vertex) | title, file_path, component_id | fallback маскировал невидимые записи 3 недели |
| Sprint | status_raw, priority, pr_refs, valid_from | name, context_md; `outcome_md` — **мёртвое поле** (никто не читает) | priority-запись была невидима всегда |
| Task | status_raw, note_md, effort_days (зеркало) | те же поля как денорм | двойная запись вручную в каждом эндпоинте |
| QG / Decision / Doc | — | всё | hist-типов нет/не используются |
| Release | статусные | всё остальное | 7 hist-строк шарятся между multi-repo вершинами |

**Инциденты, определившие оценку** (все починены, но важен генезис):
1. Тихий no-op записи тела ADR на 59/83 легаси-строк (`WHERE state_uid=null` → 0 строк при `ok:true`).
2. LH-44 wipe-класс в **10 эндпоинтах**: частичный вызов затирал непереданные поля (включая сброс `promoted→pending` у рекомендаций и стирание кураторских `content_md` гейтов).
3. Невидимые записи (vertex-only при hist-читающем слайсе): spec body, sprint priority.
4. Двойной bulk-ingest → 156 сущностей с двумя «открытыми» строками (недетерминированное чтение `[0]`).
5. `CREATE EDGE` в пустой target — тихий no-op с ложным «added».
6. ArcadeDB-ловушки: `GROUP BY @out,@in` возвращает ложные дубли; `_` в LIKE — wildcard; `DELETE VERTEX` не работает; schemaless → null-ключи прошли без ошибок.

## 1. GOAL/KAOS — дерево целей и препятствия

### Дерево целей

```
G0. LORE — единственный достоверный источник памяти проекта
├── G1. Текущее состояние читается быстро и однозначно        [✅ достигнута]
│      слайсы + открытая hist-строка; после ремонта multi-open — детерминизм
├── G2. История изменений не теряется                          [🟡 частично]
│      статусы — да (SCD2-переходы); ТЕЛА документов — нет (amend-in-place)
├── G3. Запись безопасна и честна                              [✅ после 07-02]
│      partial-safe (LH-44 sweep), body_written/linked-сигналы
├── G4. Связи целостны (граф — источник правды по связям)      [🟡 частично]
│      рёбра есть на всё, но денорм-поля (component_id, sprint_id) живут параллельно
├── G5. Знание находимо                                        [✅ после 07-02]
│      кросс-типовой ILIKE-search по id+title+body
└── G6. Инварианты хранения соблюдаются АВТОМАТИЧЕСКИ          [🔴 НЕ достигнута]
       сегодня они соблюдаются дисциплиной автора каждого эндпоинта
```

### KAOS-препятствия (obstacles) и ответственные агенты

| # | Препятствие (анти-цель) | Бьёт по | Агент-ответственный | Статус митигации |
|---|---|---|---|---|
| O1 | Автор нового write-эндпоинта не знает/забывает контракт «hist-поле → писать в hist» | G6→G3 | backend-разработчик (человек/LLM) | 🔴 только конвенция в CLAUDE.md + память сессий |
| O2 | Безусловный `SET` затирает непереданное | G3 | backend | ✅ выметено 07-02, но **ничто не мешает регрессу** в новом коде |
| O3 | Тихие no-op (`ok:true` при 0 записей) | G3, G0-доверие | backend + ArcadeDB | 🟡 body_written/linked добавлены точечно; нет общего правила «каждая запись возвращает счётчик» |
| O4 | Повторный bulk-ingest плодит дубли открытых строк | G1 | ingest-скрипты | 🔴 нет UNIQUE-гарантии «одна открытая строка на родителя» |
| O5 | Schemaless: null/битые ключи проходят молча | G0 | ArcadeDB DDL | 🟡 ключи почищены; NOT NULL/UNIQUE объявлены не везде |
| O6 | Денорм-поле и ребро расходятся | G4 | backend (write-through не гарантирован) | 🟡 слайсы COALESCE'ят, но расхождение не детектится |
| O7 | Правка тела уничтожает предыдущую версию тела | G2 | архитектура SCD2-«открытой строки» | 🔴 осознанный компромисс, нигде не зафиксирован как решение |

**Вывод KAOS:** G1/G3/G5 закрыты; конституционная дыра — **G6**: система полагается на то, что каждый будущий автор перечитает всю сагу этой сессии. O1 — материнское препятствие, порождающее O2/O3/O6.

## 2. ТРИЗ — противоречия и их разрешение

### П1. «История тел» vs «не плодить строки»
**Если** каждое сохранение тела создаёт новую hist-строку → история полная, **но** typo-фиксы и итеративные LLM-amend'ы (по 5–10 на документ за сессию) раздувают hist в мусорную ленту.
**Если** amend-in-place (сейчас) → компактно, **но** G2 нарушена: вчерашнюю редакцию решения не восстановить.

**ИКР:** история есть, мусора нет.
**Приём — разделение во времени + по условию:** чекпоинт-версионирование. Открытая строка мутируется **в пределах рабочего окна**; новая строка создаётся по **событию-границе**: смена статуса (уже так!), явный `checkpoint=true` в вызове, или смена автора/сессии. Плюс дешёвый вариант-минимум: перед каждым UPDATE тела писать предыдущее значение в лог-поле `prev_hash`/`amended_at[]` — след без полной копии.
**Решение:** зафиксировать ADR «Body versioning = checkpoint-based», добавить опц. параметр `checkpoint` в `lore_update_adr/spec`. Сложность S.

### П2. «Быстрое чтение» (денорм на вершине) vs «один источник правды» (рёбра)
**Если** убрать денорм-поля → слайсы всегда ходят по рёбрам, правда одна, **но** каждый список платит traversal, а простые `WHERE component_id=:c` ломаются.
**Если** держать денорм (сейчас) → быстро, **но** O6: расхождение никем не ловится.

**ИКР:** денорм есть, расхождения невозможны.
**Приём — посредник + самообслуживание:** денорм пишет **только** write-слой в той же операции, что и ребро (write-through), а QG-рутина еженедельно сверяет `field == edge-derived` и репортит дрифт как FAIL-метрику. Не убирать денорм — узаконить его как materialized view с контролем.
**Решение:** инвариант-запросы уже написаны в аудите 07-02 → оформить в `qg-lore-storage` (см. TOC). Сложность S.

### П3. «Гибкость schemaless» vs «целостность ключей»
**Если** объявлять жёсткий DDL на всё → миграции на каждый чих, теряем главное удобство ArcadeDB.
**Если** schemaless (сейчас) → null-ключи и молчаливые опечатки полей.

**ИКР:** жёсткость только там, где ломается идентичность.
**Приём — разделение в пространстве:** минимальный «скелет жёсткости» — NOT NULL (mandatory) + UNIQUE только на бизнес-ключи (`adr_id`, `sprint_id`, `task_uid`, `spec_id`, `release_uid`, `state_uid`…), всё остальное — schemaless как есть.
**Решение:** дополнить `LoreSchemaInitializer` + `create-schema.sql` mandatory/unique на ~14 ключей. Сложность S. ⚠️ Помнить кейс `release_id` — глобально неуникален by design, уникален только `release_uid`.

### П4 (мета). «Скорость добавления тулов» vs «соблюдение инвариантов»
Главное противоречие, оно же TOC-конфликт — см. ниже. Приём — **вынесение** (инвариант из головы автора в код-посредник).

## 3. TOC — теория ограничений

### Нежелательные явления (UDE) → дерево текущей реальности (CRT)

```
UDE: тихие no-op записи          UDE: затёртые поля       UDE: невидимые записи
UDE: дубли открытых строк        UDE: null-ключи          UDE: ложные "added"
        ▲                            ▲                          ▲
        └────────────┬───────────────┴──────────────────────────┘
                     │
   Промежуточная причина: каждый write-эндпоинт — рукописный SQL,
   повторяющий один и тот же ритуал (upsert вершины → найти открытую hist →
   динамический SET → зеркала денорма → рёбра) С НУЛЯ
                     │
   ┌─────────────────▼──────────────────────────────────────────┐
   │ КОРНЕВОЙ КОНФЛИКТ (ограничение системы):                   │
   │ инварианты хранения существуют как ЗНАНИЕ (конвенция,      │
   │ CLAUDE.md, память сессий), а не как МЕХАНИЗМ (код/DDL/QG). │
   └────────────────────────────────────────────────────────────┘
```

Подтверждение эмпирикой: один и тот же баг-класс (LH-44) был независимо воспроизведён в **10 эндпоинтах**, писавшихся в разное время разными сессиями. Это не серия случайностей — это свойство ограничения.

### Грозовая туча (evaporating cloud)

```
Цель: LORE быстро растёт И остаётся достоверным
   ├─ Требование B: быстро добавлять тулы/эндпоинты ──► Действие D:  писать прямой SQL в каждом эндпоинте
   └─ Требование C: инварианты никогда не нарушаются ─► Действие D': всё через тяжёлый централизованный слой
                                D ⟷ D' конфликт
```
**Инъекция (разрушение тучи):** D и D' не противоречат, если «централизованный слой» — это **не фреймворк, а 3 маленьких helper'а** в том же классе: `upsertVertex(type, key, fields)` (динамический SET), `amendOpenHist(type, parentKey, fields)` (поиск по @rid + LH-44 + backfill uid), `linkEdge(type, from, to, verify=true)` (с честным linked-счётчиком). Новый эндпоинт = 3 вызова вместо 60 строк SQL — **быстрее**, чем сейчас, и инвариант физически не обходится.

### Пять фокусирующих шагов

1. **Найти ограничение:** контракт-как-знание (выше).
2. **Использовать (exploit, без инвестиций):** аудит-запросы 07-02 уже существуют → оформить как QG-рутину **`qg-lore-storage`** (метрики: multi_open_count=0, orphan_hist=0, null_key=0, denorm_drift=0, dup_edges=0 c клиентским дедупом). Стоимость ~час, ловит регресс любого будущего автора.
3. **Подчинить всё остальное:** правило для новых write-эндпоинтов — «через helpers или не мержится» (ArchUnit-проверка: запрет literal `"UPDATE Know"` вне helper-класса — вписывается в существующий QG-ARCH-ENFORCEMENT).
4. **Расширить (elevate):** сами helper'ы (рефакторинг ~30 эндпоинтов постепенно, по мере касания) + DDL-скелет ключей (П3).
5. **Вернуться к шагу 1:** после helpers ограничением станет, вероятно, отсутствие as-of-T чтения (time-rewind) — уже запланировано в SPRINT_LOOM_TIMEREWIND, не трогать раньше времени.

## 4. Вердикт и дорожная карта

**Устраивает ли структура? Да — с одной конституционной оговоркой.**
Фундамент здоровый и проверен боем: граф + SCD2-открытая-строка + whitelisted-слайсы + единая точка записи + MCP — правильная топология для «памяти проекта», масштаб (≈6k вершин) даёт запас на порядки. Все острые дефекты данных и кода выметены. Но система **держит инварианты дисциплиной, а не механизмом** — каждый новый автор кода будет наступать на LH-44/vertex-vs-hist, пока контракт не станет исполняемым.

| Приоритет | Действие | Закрывает | Сложность |
|---|---|---|---|
| **Must** | QG-рутина `qg-lore-storage` из аудит-запросов 07-02 | O3, O4, O6 (детекция) | S (~1 ч) |
| **Must** | DDL-скелет: mandatory+UNIQUE на 14 бизнес-ключей (+`state_uid`) | O5, O4 (частично) | S |
| **Should** | Helpers `upsertVertex`/`amendOpenHist`/`linkEdge` + ArchUnit-запрет прямого UPDATE вне них | O1, O2 — корень | M |
| **Should** | ADR «Body versioning = checkpoint-based» + параметр `checkpoint` | O7 / G2 | S |
| **Could** | Расшить 7 шаренных ReleaseHist-строк; удалить мёртвое `outcome_md` или начать читать | гигиена | S |
| **Won't (пока)** | Полная история тел (строка на каждый amend), as-of-T reads, FT-индекс поиска | преждевременно на текущем масштабе | — |

**Ключевая мысль:** проблема LORE никогда не была в модели данных — она в том, что модель живёт в головах. Два Must-шага переносят её в машину за один день работы.
